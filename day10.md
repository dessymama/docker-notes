🐳 Day 10 – Docker Volumes (Data Persistence)
📌 Overview
In Day 10, I learned about Docker volumes and how they are used to persist data generated by containers. I understood that containers are temporary by default, and without volumes, any data created inside a container is lost when the container is removed.
📚 What I Learned
Containers are ephemeral (temporary) by default
Data inside a container is lost when it is deleted
Docker volumes are used to store data permanently
Volumes exist outside the container lifecycle
They are commonly used for databases and applications that need persistent storage
🧠 Key Concepts
Image → Blueprint
Container → Running instance
Volume → Persistent storage layer
⚙️ Important Commands
Create a volume
docker volume create my_volume
List volumes
docker volume ls
Run container with volume
docker run -d -v my_volume:/app nginx
Inspect volume
docker volume inspect my_volume
🚨 Challenges Encountered
Initially assumed data inside containers is permanent
Confused where Docker stores persistent data
Did not understand difference between container storage and volumes
Needed clarity on real-world usage of volumes
🛠️ How I Resolved It
Tested container deletion and confirmed data loss
Used Docker volumes to store data outside container lifecycle
Verified that volume data remains even after container is removed
Understood real-world use cases like databases and logs
🎯 Final Result
Successfully understood and practiced Docker volumes. Learned how to persist data in containers, which is essential for production applications.
💡 Key Lesson
Docker containers are temporary. To preserve important data, you must use volumes.
🚀 DevOps Insight
In real production systems:
Databases (MySQL, PostgreSQL) use volumes
Logs are stored in volumes
Application uploads are stored in volumes
Without volumes → data loss on container restart
